Cards are the core offensive and defensive tool in Beast Card Clash. Each card you play during your turn represents an elemental force, and the values on those cards determine whether you deal damage, survive an attack, or eliminate an opponent in the Referee phase.

## What a card is

Every card in the game carries exactly two properties:

- **Element** — one of the five elemental types (Air, Earth, Energy, Fire, or Water), or `NONE` for untyped cards. See [Elements](/mechanics/elements) for full details.
- **Value** — an integer from **1 to 10** representing the card's power. Higher values generally win comparisons in the [Referee phase](/mechanics/battle#referee).

```gdscript card.gd
# Core card data properties
var element: GameConstants.Elements
var value: int  # Range: 1–10

# Note: the constant is spelled MAX_CARD_VAlUE (with capital A) in game_constants.gd
const MAX_CARD_VAlUE = 10
```

<Note>
The `Card` data object (returned by `get_abstract_card()`) is a lightweight resource separate from the visual `CardScene` node. Use the data object when you need to pass card information between systems without the overhead of the full scene.
</Note>

## Card states

A card can be in one of three interactive states at any given time:

<Columns cols={3}>
  <Card title="Normal" icon="eye">
    The card is visible and interactive. The player can hover over it and click it to select it.
  </Card>
  <Card title="Hidden" icon="eye-slash">
    The card is face-down. Its element and value are not visible. Use this for opponent hand cards or unrevealed deck cards.
  </Card>
  <Card title="Disabled" icon="ban">
    The card is grayed out and non-interactive. Use this when a card cannot legally be played during the current turn.
  </Card>
</Columns>

These states map directly to two boolean properties on `CardScene`:

```gdscript card_scene.gd
hide_card()     # Flips the card face-down (hidden state)
disable_card()  # Grays out the card and blocks input (disabled state)
```

## Selecting a card

When you click a card that is visible and enabled, the card emits the `card_selected` signal:

```gdscript
signal card_selected(card: Card)
```

The signal passes the lightweight `Card` data object — not the scene node — so receiving systems do not need a reference to the visual card. The [Battle system](/mechanics/battle) listens to this signal during the **Turn** phase to register the card you chose to play.

<Tip>
If you are building a custom UI overlay, connect to `card_selected` on each `CardScene` instance rather than polling card state every frame. This keeps your code event-driven and avoids unnecessary per-frame checks.
</Tip>

## Visual behavior

`CardScene` extends `TextureButton` and includes three built-in visual effects that run automatically:

| Effect | Trigger | Description |
|---|---|---|
| Hover animation | Pointer enters the card | Card lifts slightly to indicate interactivity |
| Flip animation | `hide_card()` / reveal | Card rotates on its Y axis to reveal or conceal its face |
| Wave oscillation | Idle in hand | Cards gently bob in a wave pattern while sitting in the player's hand |

These effects are purely cosmetic and do not affect gameplay state.

## The CardsList resource

Card textures are not hardcoded onto individual card nodes. Instead, a `CardsList` resource maps every `element + value` combination to the correct texture. When a `CardScene` is instantiated and its properties are set, it queries `CardsList` to load the appropriate artwork.

This means adding or replacing card art only requires updating the `CardsList` resource — no changes to scene files or card scripts are needed.

## Setting card properties in code

Use `set_properties()` to assign multiple card properties at once instead of setting each one individually:

```gdscript
# Set element and value in a single call
card_scene.set_properties({
    "element": GameConstants.Elements.FIRE,
    "value": 7
})
```

To retrieve a lightweight data object from an existing `CardScene` — for example, to pass to the battle system — call:

```gdscript
var card_data: Card = card_scene.get_abstract_card()
```

## Deck and hand management

The player's deck is created during the **Start** phase of the battle state machine by `BattleManager.setup_player()`. The deck emits a `deck_updated` signal whenever cards are drawn or played, which the battle UI subscribes to in order to refresh the hand display.

Bot decks are generated automatically by `BattleManager.setup_bots()` — each bot receives a randomly composed deck at the start of every match.

<AccordionGroup>
  <Accordion title="How cards are loaded from the deck">
    When the battle starts, `setup_player()` creates a `Deck` resource for the human player and connects the `deck_updated` signal to the battle UI. As you draw cards, the UI receives the updated hand array and re-renders each `CardScene` with the correct element and value pulled from `CardsList`.
  </Accordion>
  <Accordion title="What happens to a card after it is played">
    Once you select and play a card during your turn, it is passed to the Referee state at the end of the round. The Referee compares all played cards, applies damage or deaths, and the played card is removed from your hand. See [Battle](/mechanics/battle) for the full resolution sequence.
  </Accordion>
  <Accordion title="Can a card value exceed 10?">
    No. `MAX_CARD_VAlUE = 10` (as spelled in the source) is enforced at the constant level. Cards range from 1 to 10 inclusive.
  </Accordion>
</AccordionGroup>
